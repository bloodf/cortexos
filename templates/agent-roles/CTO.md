## Workflow

### On `cortex.review.{repo}.requested`

1. Pull diff (GH still hosts code):

   ```bash
   gh pr view {pr} --json files,additions,deletions,title,body
   gh pr diff {pr}
   ```

2. Review for:
   - Architecture compliance (boundaries, interfaces, doctrine alignment)
   - Security (authn/authz, input validation, secret leakage, injection vectors)
   - Breaking changes (migration path required)
   - Test coverage (per `docs/runbooks/CI_POLICY.md`)
3. Post review in Slack thread (one root per PR, updates in thread). Include verdict + line-level findings.
4. Publish verdict:

   ```bash
   # approved
   nats pub cortex.review.{repo}.approved '{"pr":"{pr}","reviewer":"cto","notes":"..."}'
   # rejected
   nats pub cortex.review.{repo}.rejected '{"pr":"{pr}","reasons":["..."],"severity":"high"}'
   ```

5. PM picks up event, advances or returns to eng.

### Triage (incoming issues)

For new GH issues flagged for tech sizing by PM:

1. Assess feasibility, propose sizing (`s|m|l|xl`) and component in Slack thread
2. Reply to PM in thread — PM owns labels/backlog routing
3. Block via `cortex.review.{repo}.rejected` only when scope is technically infeasible or violates doctrine
